# ThreadSanitizer suppressions for the in-process V8 JS host (issue #76 / PR #82).
#
# WHY THIS FILE EXISTS
# The V8 engine comes from the SHA-pinned rusty_v8 prebuilt (tools/v8-prebuilt.json). That
# archive is denoland/rusty_v8's DEFAULT `release` static lib — it is NOT built with
# -fsanitize=thread, so V8's own code is UNINSTRUMENTED. When the multi-threaded default
# platform (v8__Platform__NewDefaultPlatform, see v8_engine.cpp) spins up its
# DefaultWorkerThreadsTaskRunner pool, TSan reports data races on memory that V8 manages under
# its OWN mutexes — mutexes whose instrumented accesses TSan cannot see inside the uninstrumented
# archive. These races are internal to V8's worker-thread pool and never touch Context-Engine
# code. They are benign reporting artifacts of mixing an instrumented embedder with an
# uninstrumented prebuilt, NOT real defects in our code.
#
# We MUST keep the multi-threaded platform: the single-threaded variant has no background task
# runner, so Isolate teardown (isolate_->Dispose() -> MemoryPool::PostDelayedReleaseTask) null-
# deref-SEGVs on every V8 CI leg. So the race is silenced here, ORTHOGONALLY, scoped ONLY to V8's
# own platform/threading internals. We do NOT globally disable TSan and we do NOT suppress any
# race in Context-Engine's own code (namespace context::), which stays fully instrumented.
#
# Wired into EVERY ctest whose executable links the V8-backed context_js (js-test_js_engine,
# js-test_gc_discipline, js-test_gc_state_hash, ts-test_ts_in_v8, ts-test_ts_debug,
# ts-test_hot_api, cli-test_profile_command), and ONLY under the `tsan` preset, via
# TSAN_OPTIONS=suppressions=<abs path> (each registering CMakeLists.txt, guarded by CONTEXT_TSAN).
# A NEW test that boots the V8 host MUST add the same wiring, or the pool races below fail it.
#
# Format: one rule per line, `<type>:<substring-pattern>` (substring match on the demangled
# frame / module name). Ref: https://github.com/google/sanitizers/wiki/ThreadSanitizerSuppressions
#
# V8's multi-threaded worker pool (the task runner + its worker threads) — the reported racer.
race:v8::platform::DefaultWorkerThreadsTaskRunner
race:DefaultWorkerThreadsTaskRunner
# The whole V8 libplatform threading layer (task runners / worker threads) is uninstrumented.
race:v8::platform::
# The whole v8::base:: support layer (thread primitives, page/region bookkeeping, containers) is
# likewise archive-internal. Observed racer under the M6 X1 GC-discipline workload (PR #189, CI
# run 29196654116): v8::base::RegionAllocator::Split (V8 worker) vs ::TrimRegion (main thread,
# inside a forced GC) — V8's page-region bookkeeping, synchronized by V8's own base::Mutex that
# TSan cannot see inside the archive. Subsumes the earlier, narrower v8::base::Thread rule.
race:v8::base::
# The uninstrumented rusty_v8 prebuilt runs V8's background worker pool CONCURRENTLY with
# main-thread Isolate teardown (~V8Engine, v8_engine.cpp:102). Those accesses ARE synchronized by
# V8's own mutexes, but TSan cannot see that synchronization inside the uninstrumented archive, so
# it reports a benign teardown race on whichever V8-internal structure is released next (observed
# across runs: v8::internal::MemoryPool::PoolImpl<PooledPage>::ReleaseUpTo page-pool release; the
# std::__Cr hash table of v8::internal::Cancelable* in the task manager; etc.). We suppress the
# ENTIRE uninstrumented-prebuilt boundary rather than chase them one-at-a-time (whack-a-mole):
#   - std::__Cr:: is Chromium's PRIVATE bundled libc++ (rusty_v8 default use_custom_libcxx) — it
#     appears ONLY inside the V8 prebuilt. Context-Engine's own instrumented code uses the system
#     libstdc++ (std::__cxx11 / plain std::), so this can NEVER mask a race in namespace context::.
#   - v8::internal:: and cppgc:: are V8's own internals / C++ GC, likewise uninstrumented.
# `race:` suppresses a report when EITHER racing stack matches, so one of these frames on either
# side of a benign V8-internal teardown race silences it.
race:std::__Cr::
race:v8::internal::
race:cppgc::
# heap::base:: is V8's shared heap utility namespace (heap::base::Worklist — the marking /
# scavenging worklists) and absl:: is the abseil copy BUNDLED inside the rusty_v8 prebuilt; both
# exist only inside the archive (Context-Engine uses neither namespace), so like std::__Cr:: they
# can never mask a race in namespace context::. Both observed as frame-1 racers under the heavier
# `context profile gc` workload (cli-test_profile_command, CI run 29196654116).
race:heap::base::
race:absl::
# Belt-and-suspenders: match any frame attributed to the rusty_v8 module (harmless no-op under
# the static-archive link where symbols land in the test executable rather than a DSO).
called_from_lib:librusty_v8
